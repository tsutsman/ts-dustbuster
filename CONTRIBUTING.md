# Внесок у проєкт ts-dustbuster

Дякуємо за інтерес до розвитку ts-dustbuster! Цей документ описує стандарти коду, процес внесення змін та вимоги до комітів.

## Код стандартів

- **Стиль коду**: Використовуйте Prettier з налаштуваннями з `.prettierrc.json`. Запустіть `npm run format` перед комітом.
- **Лінтер**: ESLint налаштований на стандарти Node.js. Запустіть `npm run lint` для перевірки.
- **Мовні стандарти**: Код має бути зрозумілим для української та англійської аудиторії. Повідомлення CLI та GUI використовують систему i18n (`src/locales/`). Нові тексти додавайте в обидва мовні файли (`uk.json`, `en.json`).
- **Коментарі**: Пишіть коментарі англійською мовою для технічних пояснень. Укр. коментарі дозволені для контексту.
- **Імена змінних**: Використовуйте camelCase для JavaScript. Імена функцій мають бути описовими.

## Процес розробки

1. **Форкніть проєкт** створіть гілку для вашої функції/виправлення.
2. **Встановіть залежності**: `npm ci`.
3. **Робота над змінами**: пишіть код, дотримуючись стандартів.
4. **Тестування**: запустіть `npm run test:coverage` та переконайтеся, що покриття не знижується. Бажано ≥80%.
5. **Лінтер та форматування**: `npm run lint` та `npm run format:check`.
6. **Коміт**: використовуйте зрозумілі повідомлення. Наприклад:
   - `feat(cli): add --dry-run support for preview mode`
   - `fix(gui): handle server errors gracefully`
   - `docs(uk): update configuration guide`
7. **Push** та створіть Pull Request (PR) у гілку `develop` (або `main` для невеликих виправлень).
8. **CI**: переконайтеся, що всі GitHub Actions дійшли успішно.
9. **Рецензування**: очікуйте коментарі від maintainer'а. Внесіть необхідні правки.

## Вимоги до тестів

- Юніт-тести для нової бізнес-логіки обов'язкові.
- Тести мають бути ізольованими, використовувати тимчасові директорії (`fs.mkdtempSync`).
- Для модифікацій GUI додайте інтеграційні тести (наприклад, перевірка відповіді сервера).
- Покриття коду вимірюється через `nyc`. Не дозволяйте значному зниженню.

## Безпека

- Не вводьте секрети у код. Використовуйте змінні середовища.
- GUI не має автентифікації — не розгортайте у відкритих мережах.
- При роботі з файловою системою завжди використовуйте `normalizePath` та перевірки `isWithin` для запобігання path traversal.
- Підтримуйте whitelist критичних системних директорій; не видаляйте без явного підтвердження.

## Структура проєкту

- `cleaner.js` — точка входу для CLI.
- `src/cli/` — парсер аргументів та запуск CLI.
- `src/core.js` — основна логіка очищення.
- `src/config.js` — завантаження та валідація конфігурацій.
- `src/locales/` — файли перекладів.
- `src/utils/` — допоміжні утиліти.
- `gui.js` — простий HTTP-сервер для веб-інтерфейсу.
- `public/index.html` — фронтенд GUI.

## Релізи

- Версії визначаються у `package.json`.
- Перед релізом переконайтеся, що CI проходить, тести покривають ключові сценарії, а документація оновлена.
- Використовуйте теги Git для версій: `git tag v1.2.3`.
